# Design: ubiquitous-language-skill

## Chosen Direction

Build `ubiquitous-language` as a standalone user-level dot-agent skill.

The skill installs through dot-agent setup. Repo-local injection happens only
when the skill is called in a repo. Default repo artifact:

```text
docs/UBIQUITOUS_LANGUAGE.md
```

Default repo instruction pointer:

```markdown
- Shared language: read `docs/UBIQUITOUS_LANGUAGE.md` when planning, implementing, or reviewing domain terminology.
```

For dot-agent itself, add the same progressive-disclosure idea to root
`AGENTS.md` and create `docs/UBIQUITOUS_LANGUAGE.md`.

## Relevant Principles

- Source repo first: edit `~/.dot-agent`, then run `setup.sh` to propagate
  runtime payloads.
- Project-local files stay project-local: `setup.sh` should not patch active
  repos.
- Progressive disclosure: one always-on instruction line should point to the
  deeper language artifact.
- Repo contracts win: default to `docs/UBIQUITOUS_LANGUAGE.md`, and if a repo
  explicitly forbids that path, stop and ask.
- Keep first implementation simple: deterministic path/inventory mechanics,
  model-curated definitions.
- Implementation alignment is the success metric.

## Decisions

### D1. Skill Ownership

Decision: create `skills/ubiquitous-language/` with `SKILL.md`, `skill.toml`,
and `scripts/ubiquitous-language.py`.

Rationale: the capability must be usable in any repo and consumed by multiple
skills. `spec-new-feature` and `review` are consumers, not owners.

Affected areas:

- `skills/ubiquitous-language/SKILL.md`
- `skills/ubiquitous-language/skill.toml`
- `skills/ubiquitous-language/scripts/ubiquitous-language.py`
- `skills/README.md`

### D2. Install Path

Decision: rely on existing `setup.sh` skill discovery. Do not add custom setup
repo injection.

Rationale: `setup.sh` already installs skills from `skill.toml`, and current
policy says setup reports project-local drift but does not patch project repos.
Repo injection belongs to the skill when explicitly invoked.

Affected areas:

- `setup.sh` verification only unless implementation discovers a real gap.
- `skill.toml` must target both `claude` and `codex`.

### D3. Repo Artifact Path

Decision: default to `docs/UBIQUITOUS_LANGUAGE.md` in every repo.

Rationale: user direction favors docs, same place for all repos. This satisfies
`semi-stocks-2/docs/doc-contract.md` and avoids root-doc sprawl.

Affected areas:

- `docs/UBIQUITOUS_LANGUAGE.md` in dot-agent.
- `/Users/ash/Documents/2026/semi-stocks-2/docs/UBIQUITOUS_LANGUAGE.md` as the
  example target when explicitly executed.

### D4. AGENTS.md Pointer

Decision: skill `init`/`apply` mode adds one progressive-disclosure line to
repo `AGENTS.md` when missing. First version does not automatically patch
`CLAUDE.md`.

Rationale: user explicitly asked for an `AGENTS.md` single line. Keeping
`CLAUDE.md` out of automatic first-wave mutation avoids broad instruction
surface edits; future alignment can compose with `improve-agents-md`.

Affected areas:

- dot-agent root `AGENTS.md`.
- repo-local `AGENTS.md` when the skill is invoked with write mode.

### D5. Artifact Schema

Decision: generate a compact Markdown file optimized for both humans and
agents.

Proposed shape:

```markdown
# Ubiquitous Language

## Purpose
<one paragraph>

## Terms
| Term | Definition | Use When | Compress/Aliases | Avoid | Evidence |
|------|------------|----------|------------------|-------|----------|

## Open Language Questions
| Question | Why It Matters | Evidence Needed |
|----------|----------------|-----------------|

## Refresh Notes
<generated metadata and source inventory summary>
```

Rationale: this keeps definitions scannable, compresses aliases, and gives
review/planning a concrete evidence trail without making a heavy glossary.

### D6. Term Scope

Decision: focus on repo domain terms. Include harness workflow terms only when
the repo's domain is dot-agent itself.

Rationale: semi-stocks should define semiconductor research/canonical pipeline
language, not generic harness internals. dot-agent should define harness terms.

### D7. Evidence Model

Decision: keep evidence simple. One authoritative source is enough.

Rationale: user direction favors simplicity. The artifact is generated by the
skill, not a locked human-authored source.

### D8. Alias Handling

Decision: compress aliases aggressively into preferred terms.

Rationale: the main goal is reducing verbose translation and implementation
misalignment.

### D9. Path Resolution

Decision: if a requested repo path is missing, fail clearly and show nearby
suggestions; do not silently alias stale names.

Rationale: the user was unsure about stale `semi-stocks` aliasing. Silent path
rewrites risk mutating the wrong repo.

### D10. Skill Integrations

Decision: update `spec-new-feature` and `review` to consume the language doc.

- `spec-new-feature`: before drafting `01_spec.md`, check for
  `docs/UBIQUITOUS_LANGUAGE.md` and use it to preserve repo terminology.
- `review`: for docs/instructions/architecture/domain changes, check
  terminology drift against `docs/UBIQUITOUS_LANGUAGE.md` when present.

Rationale: these are the places where language alignment directly improves the
implementation path.

### D11. Commands

Decision: first helper command set:

```text
ubiquitous-language.py locate [--repo PATH]
ubiquitous-language.py init [--repo PATH] [--write] [--force]
ubiquitous-language.py inventory [--repo PATH]
ubiquitous-language.py lint [--repo PATH]
```

Behavior:

- `locate`: print repo root, target doc path, AGENTS path, and status.
- `init`: create/update `docs/UBIQUITOUS_LANGUAGE.md` scaffold and optionally
  inject the `AGENTS.md` pointer.
- `inventory`: list candidate source files and ignored paths.
- `lint`: report missing language doc, missing AGENTS pointer, stale evidence
  paths, and empty required sections.

The agent remains responsible for curating the actual term definitions from
inventory evidence in the first version.

## Open Risks

- Automatic `CLAUDE.md` alignment is not included; if this causes practical drift,
  add a second-wave task through `improve-agents-md`.
- A generated doc can still become stale if users do not run refresh/lint.
- `review` terminology checks need to stay scoped; line-level code reviews
  should not become glossary audits.
- First version will not prove shorter answers directly; it targets
  implementation alignment and reviewable term evidence.

## File Map

- `AGENTS.md`: add one progressive-disclosure pointer for dot-agent's own
  language artifact.
- `docs/UBIQUITOUS_LANGUAGE.md`: generated dot-agent harness language artifact.
- `skills/ubiquitous-language/SKILL.md`: workflow and composition contract.
- `skills/ubiquitous-language/skill.toml`: install targets.
- `skills/ubiquitous-language/scripts/ubiquitous-language.py`: locate/init/
  inventory/lint helper.
- `skills/README.md`: add skill to workflow/composition docs.
- `skills/spec-new-feature/SKILL.md`: language doc pre-read rule.
- `skills/spec-new-feature/claude/SKILL.md`: matching runtime entry update if
  needed.
- `skills/spec-new-feature/codex/SKILL.md`: matching runtime entry update if
  needed.
- `skills/review/SKILL.md`: terminology drift review rule.
- `skills/review/codex/SKILL.md`: matching runtime entry update if needed.
- `docs/artifacts/ubiquitous-language-skill/`: planning/design/tasks.
